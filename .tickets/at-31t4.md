# Design outfit generation pipeline

Research and design the outfit generation process. Determine how to composite multiple clothing items onto model. Design layering system for inside/outside tops. Create workflow using Kolors and Bria generators. This is the unresolved design issue from PRD Section 11.1.

## Design Decisions (Resolved)

### Strategy: Sequential Application

The Kolors Virtual Try-On API (`fal-ai/kling/v1-5/kolors-virtual-try-on`) accepts exactly one `human_image_url` + one `garment_image_url` per call. To support multiple garments, we use **sequential application**:

1. Start with the user's model photo as `human_image_url`
2. Apply the first garment via Kolors → get result image
3. Use that result as the new `human_image_url` → apply next garment
4. Repeat for all selected garments
5. Final output is the last image in the sequence

**Trade-offs accepted:**
- Slower: N garments = N * ~15-30 seconds of Kolors processing time
- Potential quality degradation on later passes (each pass is a new generation)
- Earlier garments may be partially overwritten by later ones (mitigated by layering order)

### Slot Requirements

- **Model**: Required (always)
- **At least one clothing slot**: Required
- **All other slots**: Optional — user can leave any combination blank
- Supported slots: Inside Top, Outside Top, Bottoms, Shoes, Socks, Hat

For v1, assume Kolors handles all slot types adequately (tops, bottoms, shoes, socks, hats). Re-evaluate if test images reveal issues with specific categories.

### Layering Order

Garments are applied in an order that puts inner/covered layers first and outer/visible layers last. This way, later garments naturally cover earlier ones:

1. **Socks** (most covered by shoes)
2. **Inside Top** (base layer, partially covered by outside top)
3. **Bottoms** (pants/skirts)
4. **Outside Top** (jacket/coat — covers inside top)
5. **Shoes** (covers socks)
6. **Hat** (topmost, applied last)

Only slots with a selection are included — empty slots are skipped.

### Background Removal

- **New uploads** (camera, photo library, paste): Run through Bria Background Removal (`fal-ai/bria/background/remove`) before storing as a wardrobe item
- **Existing board items**: Already processed — do NOT re-run background removal when selecting them for an outfit

Background removal happens at **upload time**, not at generation time. By the time the outfit generator runs, all garment images should already have clean backgrounds.

### Implementation: New Composite Generator

The pipeline is implemented as a **new generator** registered in `packages/backend`, not as frontend orchestration or a separate worker actor. This keeps the generation pipeline self-contained and follows the existing generator pattern.

**Generator name:** `outfit-generator` (or similar)
**Artifact type:** `image`

#### Input Schema

```python
class OutfitGeneratorInput(BaseModel):
    model_image: ImageArtifact              # Required: user's photo
    inside_top_image: ImageArtifact | None   # Optional
    outside_top_image: ImageArtifact | None  # Optional
    bottoms_image: ImageArtifact | None      # Optional
    shoes_image: ImageArtifact | None        # Optional
    socks_image: ImageArtifact | None        # Optional
    hat_image: ImageArtifact | None          # Optional
```

At least one of the optional garment slots must be provided. Validation should reject requests with only a model and no garments.

#### Internal Flow

```
1. Resolve all input artifacts (standard artifact resolution)
2. Build ordered garment list from non-None slots (layering order above)
3. Validate at least one garment is present
4. Set current_image = model_image
5. For each garment in ordered list:
   a. Publish progress: "Applying {slot_name}... (step {i}/{total})"
   b. Upload current_image + garment to Fal
   c. Call Kolors Virtual Try-On API:
      - human_image_url = current_image
      - garment_image_url = garment
   d. current_image = Kolors result image
6. Store final current_image as the generation output
7. Return GeneratorResult with single ImageArtifact
```

#### Progress Reporting

Since this is a multi-step generation, progress should be granular:
- `"Initializing..."` (phase: initializing)
- `"Applying inside top (1/3)..."` (phase: processing, with step info)
- `"Applying bottoms (2/3)..."`
- `"Applying outside top (3/3)..."`
- `"Finalizing..."` (phase: finalizing)

The frontend generation progress UI should show which step is being processed.

#### Cost Estimation

Each Kolors call costs ~$0.05. Total cost = $0.05 * number_of_garments.

#### Error Handling

- If any intermediate Kolors call fails, the entire generation fails
- No partial results are saved (the final composite is all-or-nothing)
- Standard retry policy from the Dramatiq worker applies (max 3 retries)

### Registration

Add the new generator to `packages/backend/baseline-config/generators.yaml`:

```yaml
- class: "boards.generators.implementations.fal.image.outfit_generator.OutfitGenerator"
  enabled: true
```
